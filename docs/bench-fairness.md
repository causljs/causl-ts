# Bench fairness — boundary-semantics decisions per scenario

This document records the **write-boundary semantics** the four library
harnesses (`causl`, `jotai`, `redux-toolkit`, `mobx`) implement for each
canonical scenario in `@causljs/bench`. The decisions here are load-bearing:
without them, the comparison_table.md numbers measure transactional-loop
overhead rather than the property each scenario claims to test.

The fair-fight parity gate (`packages/bench/test/fair-fight-parity.test.ts`)
asserts every harness honors this document. Drift surfaces as a test
failure, not as silent number drift.

The **apples-to-oranges conformance gates** (#873) catch a different
class of drift — workload-shape asymmetries that survive the boundary
gate. See [Apples-to-oranges conformance invariants](#apples-to-oranges-conformance-invariants-873)
below.

## Why this matters (the bug this prevents)

Pre-#678, the bench had unintentional asymmetry on two scenarios:

- `equality-cutoff × 10000`: **causl ran 10 000 separate `g.commit` calls**;
  jotai ran 10 000 bare `store.set`; **mobx wrapped the entire loop in ONE
  `runInAction`**; redux ran 10 000 separate `dispatch`. The 4685 ms vs
  5 ms causl-vs-jotai gap was commit-loop overhead, not the cutoff. Mobx's
  0.97 ms was its single-batch coalescing.
- `scrolling-viewport × 10000`: **mobx batched the entire scroll** in one
  `runInAction`; the other three ran per-step. Mobx's "4 ms" was a
  batching artifact.

The audit's verdict: every perf experiment downstream was measuring against
numbers that didn't say what they meant. #678 fixed this (closed); this
document records the regime per scenario so future contributors don't
reintroduce it.

## The two regimes

Each scenario picks one of three regimes:

### Single-boundary regime

All four libraries wrap their N input writes in **one** outer transactional
boundary. The cell measures the property the scenario name claims (cutoff
correctness, batching coalescing) on a single transactional event with N
underlying writes — not on a transactional loop.

| Library | Boundary primitive |
|---------|--------------------|
| causl | `g.commit('label', tx => { for (...) tx.set(...) })` |
| mobx | `runInAction(() => { for (...) box.set(...) })` |
| redux-toolkit | `store.dispatch(slice.actions.bumpAll(updates))` (one batched action carrying N updates) |
| jotai | **No native batching primitive** — runs N bare `store.set` calls. Documented contract gap; results note this asymmetry explicitly. |

Scenarios in this regime: `equality-cutoff`, `batch-commit`.

The jotai gap is intrinsic to jotai's design (push-based, no batching API).
On `batch-commit` the gap is the contract being measured (causl's headline
win is "batched writes coalesce"). On `equality-cutoff` the gap is
documented and adopters can read the comparison table with the asterisk
in mind.

### Per-step regime

Each library pays one boundary per logical step. The scale parameter
drives the iteration count; every library iterates N/step times and pays
one boundary per iteration.

| Library | Per-step boundary |
|---------|-------------------|
| causl | `g.commit('scroll-${i}', tx => tx.set(cells[i], i))` |
| mobx | `runInAction(() => cells[i].set(i))` |
| redux-toolkit | `store.dispatch(slice.actions.set({ index: i, value: i }))` |
| jotai | `store.set(cells[i], i)` |

Scenarios in this regime: `scrolling-viewport`, `async-race` (fixed-step,
2 boundaries).

This regime mirrors a real React app under user interaction — every
keystroke / scroll frame / event is its own transactional unit.

### Natural-ops regime

The scenario describes a fixed number of writes (1, 2, etc.). Each library
writes that exact number of boundaries — no scale-driven loop wraps the
boundary primitive.

Scenarios in this regime: `linear-chain` (one bump), `diamond` (one bump),
`dynamic-dep-flip` (two writes — flip flag, write abandoned).

## Per-scenario decisions

| Scenario | Regime | Property being measured |
|----------|--------|------------------------|
| `linear-chain` | natural-ops, 1 bound | Forward propagation cost down a chain of N derivations |
| `diamond` | natural-ops, 1 bound | Glitch-free recomputation through a diamond DAG |
| `dynamic-dep-flip` | natural-ops, 2 bound | Abandoned-input writes don't trigger downstream recompute after a flip |
| `scrolling-viewport` | per-step, ~100 bound at scale=100 | Mount/unmount churn under user scroll; per-frame boundary cost |
| `async-race` | natural-ops, 2 bound | Late observer sees only the latest value |
| `batch-commit` | single-bound, 1 (jotai: N) | Batched-commit primitive coalesces N writes into one notification |
| `equality-cutoff` | single-bound, 1 (jotai: N) | Derivation that recomputes to `Object.is`-equal value does not fire downstream |

## Reading the comparison table

`comparison_table.md` (regenerated by `pnpm --filter @causljs/bench
bench:report`) should be read with the regime table above in mind:

- **Single-boundary scenarios** — every cell measures the cutoff /
  batching property at one logical event. Causl's win on `batch-commit`
  is real; the jotai cell is the documented gap.
- **Per-step scenarios** — every cell measures per-frame cost; mobx no
  longer benefits from a hidden whole-loop batch.
- **Natural-ops scenarios** — every cell does the exact same N writes
  with the same boundary count.

Cells whose numbers shifted dramatically post-#678 were precisely the
cells that had been paying for the bug. Those numbers are the honest
baseline every perf experiment under the now-closed #679 umbrella
measured against; they remain the baseline for the post-Rust-engine
(#1133) work picking up where #679 left off.

## How the parity gate enforces this

`packages/bench/test/fair-fight-parity.test.ts` does a structural check
against each harness's source:

1. Extracts the `case '<scenario>':` block from each library's `runOnce`.
2. Counts boundary-primitive call sites textually.
3. For single-boundary regime: asserts exactly **one** boundary call,
   not wrapped in a scale-driven `for` loop (jotai is the documented
   exception — its boundary IS in a `for` loop).
4. For per-step regime: asserts the boundary IS inside a scale-driven
   `for` loop (every library).
5. For natural-ops regime: asserts the static boundary count matches
   the expected (1, 1, 2, 2 per the table above).

Plus a functional smoke pass that runs every (library × scenario) cell
at scale=100 and asserts the runtime sample count is non-zero.

## Reintroducing asymmetry is a test failure

If a future PR changes a harness's boundary structure, the parity gate
fails before the bench numbers can lie. To intentionally change a
scenario's regime, update both:

1. The harness file (e.g., `packages/bench/src/libraries/causl.ts`).
2. The regime spec table in `packages/bench/test/fair-fight-parity.test.ts`.
3. This document.

Single-PR atomic update; reviewers see the regime change explicitly.

## Apples-to-oranges conformance invariants (#873)

The boundary-semantics gate above catches one class of drift —
"different number of transactional units." There is a second class the
A15 audit surfaced: **workload-shape drift inside the boundary**, where
two harnesses both pay the right number of boundaries but do
structurally different work inside them. Three invariants close that
gap; each ships with a conformance test under
`packages/bench/test/conformance/`.

### 1. Recompute-count parity for `linear-chain`

`packages/bench/test/conformance/recompute-count-parity.test.ts`
asserts every library's `linear-chain` harness reports
`recomputes >= scale` on the head bump at `CONFORMANCE_SCALE`.

**Why:** pre-#873, the redux harness implemented `linear-chain` as a
hand-rolled `for (i=1;i<state.length;i++) state[i]=state[i-1]+1` loop
inside a single `createSlice` reducer. One Immer mutation, no
`createSelector` chain, no derivation count. The other three libraries
(causl `derived`, jotai `atom((get) => ...)`, mobx `computed`) build
`scale` derivation nodes and recompute each one on the bump. Same
end-state, but the redux cell measured a tight reducer loop while the
others measured derivation propagation — the headline 782× redux gap
claim was partly comparing different work.

**Fix:** redux's `linear-chain` is now a chained `createSelector`
graph mirroring jotai/mobx — slice state is one `head` input, and
`scale` chained reselect derivations each add 1. The invariant test
asserts the recompute counter is at least `scale` for every library.

### 2. Cutoff-input invariant for `equality-cutoff`

`packages/bench/test/conformance/cutoff-input-invariant.test.ts`
asserts every harness's `equality-cutoff` block reads from at least
one input that mutates during the step.

**Why:** pre-#873, the redux harness defined the constant derivation as
`selectK = createSelector((_s) => 0, () => 42)`. The input selector
returns the literal `0`, so reselect's input-tuple cache hits on every
dispatch — the combiner never runs, and the "cutoff" credited to redux
was actually input-tuple memoisation, not the `Object.is`-equal
output check that the scenario name claims to test. Different
mechanism than what causl/mobx are exercising.

**Fix:** redux's `selectK` now reads every mutating cell via an array
of input selectors (`(s) => s[i]`); reselect must walk the inputs each
dispatch, the combiner runs, returns 42, and the `Object.is` output
check is what cuts off the downstream `selectD`. The invariant test
extracts the case body and asserts a textual marker proving the
constant derivation references the mutating-input collection.

### 3. Setup-time-per-cell parity for `scrolling-viewport` (FLAG-only)

`packages/bench/test/conformance/setup-time-parity.test.ts` measures
each library's setup time for `scrolling-viewport` and emits a
`console.warn` if any library is more than 2× the cross-library
median.

**FLAG-only — does NOT auto-fail the bench.** The acceptance criteria
explicitly require a warning, not a hard gate, because setup-cost
asymmetries are partly architectural (jotai mounts atoms lazily, causl
mounts eagerly at `g.input(...)`) and partly noise. The warning
surfaces drift in CI logs without breaking the merge gate; if a
follow-up PR decides to harden the flag into a gate, the test is the
canary that motivates the refactor.

## Public-numbers callout: cells that shift post-#873

The redux rewrites change the work redux is doing inside the
`linear-chain` and `equality-cutoff` cells, so the `comparison_table.md`
numbers for those cells will shift on the next regen. Specifically:

- **`redux × linear-chain` (all scales)**: previously a tight Immer
  reducer loop; now a chained-`createSelector` derivation walk. The
  recompute counter changes from `1` to `scale`. The runtime is
  expected to *increase* substantially — reselect's per-link
  memoisation walk is materially more expensive than a hand-rolled
  loop. The post-#678 published number was redux 5.33ms at scale=10000
  (vs causl skipped); post-#873 redux will report a higher number,
  which is the correct apples-to-apples cost.
- **`redux × equality-cutoff` (all scales)**: previously the
  constant-derivation input was `(_s) => 0` (input-tuple cache
  shortcut); now it reads every mutating cell. Reselect's cache walk
  now visits `scale` inputs per dispatch; the runtime is expected to
  *increase*. The published number was redux 7.30ms at scale=10000;
  post-#873 redux will report a higher number reflecting the actual
  output-cutoff cost (not the input-cache shortcut).
- The recompute counter for `redux × linear-chain × 100` in
  `packages/bench/fixtures/baseline.json` changes from `1` to `100`.
- `redux × equality-cutoff` notification + recompute counts are
  unchanged (the cutoff still works — `selectD` runs once on the cache
  miss and `Object.is`-cuts off downstream).

Causl, jotai, and mobx cells are unchanged — those harnesses already
satisfied the invariants. The headline architectural picture (causl
wins `batch-commit`; mobx leads `equality-cutoff`; etc.) is
preserved; only the redux numbers move, and they move toward the
honest baseline.
