## What I'm shipping

Conal's framing is the load-bearing one for this EPIC: **a predicate is a function from behaviors to truth values**, and the §3.1 theorems are precisely the behaviors-that-are-allowed — every `(pre, post)` pair the predicate admits is a state of affairs the adapter is willing to defend, and the predicate fails closed on the rest. The four predicates above (`originBoundResolution`, `singlePipelineMutation`, `promiseIdentityStability`, `behaviorDomain`) are total over the GraphTime line; the four counterexample fragments in §3.1 are concrete enough that fast-check shrinking can land each one as a single-line minimal repro; the four CI anchors are the gates we owe the spec.

Brendan's framing closes the runtime end: **every microtask boundary is a chance for a theorem to fail**. The `loading → loaded` transition crosses one microtask (the loader's `.then`); the `loading → stale` transition crosses one microtask plus one unrelated commit landing in the gap; the `loading → errored` transition crosses one microtask plus an exception unwound through `tx.set`'s catch arm. The property tests do not cheat by collapsing those microtask boundaries — every test settles its loaders through `Promise.resolve()` chains the test author controls, and every "unrelated commit" is a `graph.commit` call landing on a different node between the loader's settle and the resolution commit. If a theorem holds only because the test cheated on microtask ordering, the test is wrong and the theorem is unproven.

Anders's framing closes the type end: **the type-d fixtures are the mechanism `tsc` rejects the wrong shape at compile time**, and that mechanism is older and stronger than any runtime test. The two type-d fixtures (`no-tx-escape.types.ts` and `loading-arm-shape.types.ts`) are not "tests of types" in the testing-library sense — they are compile-time assertions that the public surface admits no escape route by which the runtime invariant could be falsified. A PR that adds a method whose return type contains `Tx`, or that changes the `loading` arm's `promise: Promise<unknown>` to `promise?: Promise<unknown>`, fails `tsc --noEmit` before any test runs. The runtime tests are the safety net for shapes that pass type-check; the type fixtures are the shape gate that runs first.

The shape of the deliverable is six files plus one Rust rule plus one acceptance meta-test. Four property tests, one per theorem; two type-d fixtures, one for Theorem 2's `Tx`-escape gate and one for Theorem 3's `promise`-field-shape gate. Theorem 3 also gets a React-side companion test that mounts a real `<SuspenseList>` against three resources and asserts the renderer sees pointer-identical Promises across re-renders — that is the test that closes §9.1 row 17, and it does not belong in `packages/sync` because the renderer it interrogates lives in `packages/react`. Theorem 4 needs no type-d fixture because `RetentionResult<T>` is already two-armed at the type level and `graph.readAt` already returns it; the property test is sufficient. The Rust rule for Theorem 1 (`resource:origin-bound`) lands in TASK 10.5 and depends on Schema 3; the four property tests + two type-d fixtures stand on their own and do not need it.

The acceptance gate is `packages/sync/test/theorems/all-theorems-acceptance.test.ts`, a meta-test that imports each theorem's property file + type fixture, asserts each is registered via `propertyTrials` (the §15.2 conformance walker's hook, #437), and asserts each compiles via `tsd`. The meta-test exists so that a PR deleting one of the four files fails CI loudly, with the deletion named — not silently, with the test count dropping by one and nobody noticing. This is the same gate the §15.2 conformance walker enforces for the engine's property suite, ported one layer up.

## Brutal-critical review

**Where the spec is right.** Each of the four §3.1 theorems has (a) a formal predicate written as a TypeScript signature against the `ResourceState<T>` discriminator the package actually ships, (b) a proof obligation pinned to a specific phase of the commit pipeline (Phase A's read of `graph.now`, the `tx.set(node, ...)` payload, Phase A's "throw short-circuits" closure, the engine's `readAt` machinery), (c) a falsifying counterexample fragment a reviewer can match to a PR diff at sight, and (d) a CI anchor naming the file the test lives in. The predicates are denotational in Conal's sense — they reduce a behavior to a single truth value — and they are total over the GraphTime line, which is precisely the property that makes them mechanically checkable. There is no third arm of `RetentionResult<T>`, no third branch of the loaded/stale dichotomy, no fractional GraphTime tick; the disjunction the predicate quantifies over is the disjunction `tsc` already enforces.

**Where the spec might be wrong.** The predicates quantify over the *post-state* of each transition, not over the *legality* of the transition itself. A draft that adds a sixth arm to `ResourceState<T>` — call it `'reconciling'`, with whatever payload is convenient — and threads the new tag through `assertNever` at every dispatch site will satisfy all four predicates: Theorem 1 (origin-bound resolution still holds for the four arms it remains defined on, and the new arm is its own equivalence class), Theorem 2 (the new transition still routes through `graph.commit`), Theorem 3 (the new arm can carry its own Promise if it has loading-shaped semantics, or omit one if it does not), Theorem 4 (the registration time is unchanged). And yet the chart in `docs/lifecycle.md` §1 has been silently widened, the §17 commitment 7 fence has been moved, and the closed event vocabulary the team committed to has acquired a row no review meeting authorised. The four predicates do not catch that. The five-arm DU plus `assertNever` is what catches that — every dispatch site polices a missing or extra arm, the wire schema (`schema: 2`) is closed at two top-level `kind` constants, and the `lifecycle.md` chart is the human-readable artifact a chart-edit PR has to touch alongside the code. **The §3.1 predicates are necessary but not sufficient; the chart is what they are not.** A reviewer who sees the four CI gates green and stops reading is exactly the failure mode this section warns against.

**The 1000-trial floor is not negotiable, and the failure mode if it is lowered is invisible.** A contributor who sees the property tests timing out under load and pushes a "small fix" lowering `numRuns` from 1000 to 100 makes the test pass and silently weakens four theorems by a factor of 10. The §15.2 conformance walker (`packages/core/test/spec-15.2-conformance.test.ts`, #437) already enforces the floor by walking every test that calls into the `propertyTrials` registry and asserting each registers ≥1000 trials; this EPIC's four property tests register through that same hook, so the floor is policed at the meta-level and the "small fix" fails CI before merge. We are paying the registration cost for that policing on every theorem test, deliberately.

**The Theorem 3 React companion test is the only test in this EPIC that mounts a renderer.** Every other test runs against `@causljs/core` and `@causljs/sync` directly, with no React, no JSX, no DOM. The React test is necessary because §9.1 row 17 is specifically about what React's `SuspenseList` sees across re-renders — the engine-side promise identity is the prerequisite, not the contract. We considered moving the React test out of this EPIC into a separate "renderer integration" EPIC and rejected: the only reason the engine guarantees Promise identity is the React adapter's need for it, and shipping the engine-side property test without the renderer-side test that *uses* the property is a half-finished proof. The React test goes here, in this EPIC, with its own task.

**The Rust rule for Theorem 1 is split off into TASK 10.5 because it depends on Schema 3.** That dependency is not on EPIC-1 landing first — the rule is written against the IR shape Schema 3 will publish, and the rule's tests use hand-rolled IR fixtures that match the Schema 3 shape today, but the IR walker the rule plugs into is in EPIC-1's domain and lands with EPIC-1. We could write the rule today, with its fixtures, and land it as dead code (no walker calls it); we will. The four property tests + two type-d fixtures do not depend on it — they run against runtime values, not IR — and the four theorems are gated on them today. The Rust rule is the static-fact layer that catches the same falsifications before runtime, and it is the layer that enforces the bi-implication on exported `CauslModel` values without running them. Two layers, both load-bearing, neither sufficient alone.

**One thing we explicitly are NOT shipping: a Theorem 4 type-d fixture.** `RetentionResult<T>` is already two-armed at the type level (`{ arm: 'evicted'; oldestRetainedTime: GraphTime } | { arm: 'retained'; value: T }`); `graph.readAt` already returns it; an adapter that synthesises an `idle` value for `t < registrationTime` is a *runtime* falsification that the property test catches, not a *type* falsification that `tsd` would catch. We considered a type fixture asserting "no public function whose return type is `ResourceState<T>` accepts a `t: GraphTime` argument" (which would force callers through `readAt`), and rejected: there is no such function on the public surface today, and adding the fixture as a guard against a function that does not exist is yak-shaving the type system. The runtime property test is the floor for Theorem 4.

### TASK 10.1 — Theorem 1 property test: `origin-bound-resolution.spec.ts`

> **Divergence (as of v0.9.0):** The randomised-interleaving fast-check property for Theorem 1 shipped at `packages/sync/test/properties/origin-bound-resolution.property.test.ts` (EPIC-9 / Property 2), not at the planned `theorems/origin-bound-resolution.spec.ts`. The theorems-directory witness for Theorem 1 ships as `packages/sync/test/theorems/theorem-1-origin-pinning.test.ts` — a smaller integration-level test covering sequential and same-graph fetches. The planned `_arbitraries.ts` helper file was not extracted; the generators live in `@causljs/sync-testing-internal` instead (per EPIC-9 / TASK 9.1).

**Files:** `packages/sync/test/theorems/origin-bound-resolution.spec.ts` (new). The existing `packages/sync/test/fetch-interleavings.spec.ts` (Survey 3's "1 of 4 has a dedicated property file") stays where it is; its property generator is extracted into `packages/sync/test/theorems/_arbitraries.ts` (also new) so the new spec can re-use the loader-interleaving generator without duplicating the arbitrary definitions.

Per §3.1 Theorem 1's mechanical anchor. The test generates a random interleaving of N concurrent loaders against M unrelated commits (N ∈ [1, 8], M ∈ [0, 32], drawn fresh per trial) and asserts the bi-implication on every produced `(pre, post)` pair. The bi-implication is `post.state === 'loaded' ↔ post.origin === post.loadedAt` and `post.state === 'stale' ↔ post.origin < post.loadedAt`; the third branch (`post.origin > post.loadedAt`) is structurally unreachable per `SPEC.md` §3 Theorem 4 (monotonicity), and the test asserts loudly if the generator ever produces it — that branch firing is itself a Theorem 4 falsification and the test logs the trial seed before failing.

#### TDD test suite (the file IS the suite, with multiple `it` blocks)

- **Golden: `origin === loadedAt → loaded`.** Single resource, single loader, no unrelated commits between begin and resolve. Assert post-state is `{ state: 'loaded', value, origin: t, loadedAt: t }` with `origin === loadedAt`.
- **Golden: `origin < loadedAt → stale`.** Single resource, single loader, exactly one unrelated commit landing on a different node between the loader's microtask settle and the resolution commit. Assert post-state is `{ state: 'stale', value, origin: t, loadedAt: t + 1 }` with `origin < loadedAt`.
- **Edge: `origin > loadedAt` is structurally unreachable.** The test contains an `it.skip` block documenting the case and asserting `false` if reached; the property runner refuses to generate this case because the arbitrary's `loadedAt` is computed from `graph.now` at the resolution boundary, which is monotonic, but the assertion is wired so a future change that allows retrograde GraphTime fails this case loudly.
- **Property: 1000 trials of random interleavings.** `fast-check.assert(fc.property(arbInterleaving, (trace) => predicate(trace.pre, trace.post)))`, with `numRuns: 1000` and `seed` printed on failure. Registered via `propertyTrials.register('§3.1.T1', { numRuns: 1000 })` so the §15.2 conformance walker enforces the floor.
- **Counterexample: §3.1's exact fragment falsifies the test.** A fixed scenario with `interleaved.length === 2` (two unrelated commits between begin and resolve) and `resolvesWith.ok === true` lands `state: 'stale'`, NOT `state: 'loaded'`; the test asserts the post-state is `stale` and would fail if the implementation regressed to "decide based on `performance.now()` elapsed". The `performance.now()`-based falsification fragment from §3.1 is checked into `packages/sync/test/theorems/_falsifiers/origin-bound-elapsedMs.ts.txt` (not compiled — the `.txt` extension is deliberate) so a reviewer running `grep -r 'performance.now' packages/sync/src` against a draft can match it at sight.

#### 5 concerns

1. **Floor: 1000 trials.** Registered via `propertyTrials.register('§3.1.T1', { numRuns: 1000 })`; the §15.2 conformance walker fails CI if the count is lowered or if registration is removed.
2. **Total over GraphTime line — no third branch.** The predicate is total: `origin < loadedAt`, `origin === loadedAt`, or `origin > loadedAt`, and the third case is structurally unreachable per Theorem 4. The test asserts loudly if the third case is ever generated; the assertion is wired through `expect.fail` not `assert(false)` so the failure carries the trial seed.
3. **`origin` preserved across post-state.** The test asserts `pre.origin === post.origin` on every trial — the loading arm's `origin` is the same `GraphTime` value that ends up on the resolved arm. This is the structural invariant that makes the bi-implication a function of two clocks rather than a function of "whichever clock the implementation happened to read".
4. **Counterexample fragment from §3.1 falsifies the test.** The `performance.now()`-based fragment, hand-imported into a parallel test fixture file, produces a `stale` post-state for an episode where `origin === graph.now` at resolution; the property test on the falsified implementation fails on its first 100 trials with a deterministic seed.
5. **No race condition — controlled loader settle.** Loaders are constructed via the test's `controlledPromise()` factory which exposes `resolve` and `reject` to the test body; no `setTimeout`, no `setImmediate`, no `queueMicrotask`. The test author owns the microtask ordering; Brendan's reading is enforced.

### TASK 10.2 — Theorem 2 property test + type fixture

> **Divergence (as of v0.9.0):** The Theorem 2 witness shipped as `packages/sync/test/theorems/theorem-2-single-pipeline-mutation.test.ts` — an integration-level test asserting single-Loaded-notification cardinality, not the proxy-instrumented fast-check property test the plan described. **Files:** `packages/sync/test/theorems/single-pipeline.spec.ts` (new), `packages/sync/test/theorems/no-tx-escape.types.ts` (new — type-d fixture).

Per §3.1 Theorem 2. The runtime property test instruments `graph.commit` with a counter via a `Graph` proxy and asserts that the cardinality of resource-state transitions observed via `graph.subscribe(node, ...)` over a randomised event sequence equals the cardinality of `graph.commit` calls whose `intent` matches `/^(fetch|invalidate|fail|dep-changed):/`. The type-d fixture asserts via `tsd` that the adapter's public surface — `@causljs/sync`'s exports — exposes no function whose return type contains `Tx` or any structural equivalent.

#### TDD test suite

- **Property test: cardinality equality.** Random event sequence of length L ∈ [10, 200] over K ∈ [1, 8] resource nodes; for each trial, count (a) state transitions observed by the subscriber and (b) commits whose intent matches the resource-event regex. Assert (a) === (b). 1000 trials.
- **Property test: refused transitions produce 0 commits + byte-identical pre-state.** Generate a chart-illegal event (`Idle → Errored`, `Stale → Errored`, `Errored → Errored`) and assert the commit counter does not advance and `graph.now` does not advance. Use `structuredClone(graph.snapshot())` for the byte-identical assertion.
- **Property test: every observable transition has exactly one `graph.commit` backing it.** Combines the two above into a single trial-level assertion: for each observed transition, exactly one commit fired in the immediately preceding microtask window; for each refused event, zero commits fired and the snapshot is unchanged.
- **Type-d fixture: no `Tx` in public surface.** `tsd` assertion that `import * as Sync from '@causljs/sync'; type AnyReturn = ReturnType<Sync[keyof Sync]>` does not contain `Tx` or `{ set: <T>(node: Node<T>, next: T) => void }` (the structural equivalent). A PR that adds a method like `getCurrentTx(): Tx | null` fails `tsc --noEmit` before any test runs.
- **Type-d fixture: structural equivalent gate.** `tsd` assertion that no exported function returns a value whose type includes `{ set: any; get: any; ... }` matching the `Tx` shape under structural typing — closes the "I just renamed `Tx` to `Mutator`" loophole.

#### 5 concerns

1. **Type-level enforcement — the `Tx` escape is sealed at the type system before any test runs.** The type-d fixture is the floor; the runtime counter is the safety net for shapes that pass type-check.
2. **Runtime counter — instrument `graph.commit` with a counter; assert cardinality matches state-transition count.** The instrumentation is via a `Proxy` wrapping `graph` that increments a counter on every `commit` call; the proxy is constructed per trial so counters do not leak across trials. The cardinality assertion is the bi-implication of "transitions land through commits".
3. **Throw arm — a refused transition produces 0 commits and 0 GraphTime advance.** The `ForbiddenResourceTransitionError` thrown from Phase A short-circuits the pipeline; the test asserts `graph.now` is unchanged and the counter is unchanged across the throw.
4. **Property: 1000 trials.** Registered via `propertyTrials.register('§3.1.T2', { numRuns: 1000 })`; same floor enforcement as Theorem 1.
5. **No race condition — single-threaded.** All event sequences are synchronous within a trial; loader microtasks (where present) settle via `controlledPromise().resolve()` calls the test body owns. No `setTimeout`, no `setImmediate`.

### TASK 10.3 — Theorem 3 property test + type fixture + React companion

> **Divergence (as of v0.9.0):** The Theorem 3 witness shipped as `packages/sync/test/theorems/theorem-3-promise-identity-stability.test.ts` plus the EPIC-9 fast-check property at `packages/sync/test/properties/promise-identity-stability.property.test.ts`. The `loading-arm-shape.types.ts` `tsd` type-d fixture did NOT land. The planned dedicated React `SuspenseList` companion `packages/react/test/suspense-promise-identity.spec.tsx` did NOT land as a separate file; renderer-side coverage is provided by the existing `packages/react/test/useCauslSuspense.integration.test.tsx` (which exercises a single `<Suspense>` boundary against a real renderer but does not specifically pin pointer-identical Promises across re-renders inside a `<SuspenseList>`). If §9.1 row 17 needs a sharper renderer-side gate than `useCauslSuspense.integration.test.tsx` provides, the planned companion test remains a future task.

**Files:** `packages/sync/test/theorems/promise-identity.spec.ts` (new), `packages/sync/test/theorems/loading-arm-shape.types.ts` (new — type-d fixture), `packages/react/test/suspense-promise-identity.spec.tsx` (new — React renderer integration).

Per §3.1 Theorem 3. The engine-side property test runs a fast-check generator producing N ∈ [2, 64] `graph.read(node)` observations interleaved with M ∈ [0, 16] unrelated commits during a single loading episode, then asserts every pair of `loading`-arm Promises is `===`. #### TDD test suite

- **Property test: pointer equality across reads in the same episode.** 1000 trials of N reads ∈ [2, 64] interleaved with M unrelated commits ∈ [0, 16] during a single loading episode. Assert `read1.promise === read2.promise` for every pair where both are on the `loading` arm with the same `origin`.
- **Property test: episode boundary produces fresh Promise.** Re-fetch from `Loading` (the `(re-fetch: → Loading)` cell on the chart) starts a new episode with a new `origin`; assert the new arm's `promise` is `!==` to the previous arm's promise. The predicate is per-episode, and the test asserts the boundary is observed.
- **React companion: SuspenseList sees pointer-identical Promises.** Mount `<SuspenseList revealOrder="forwards">` containing three `<Suspense>` boundaries, each wrapping a `useCauslSuspense(node)` hook against a different resource. Across three re-renders triggered by unrelated commits, capture the Promises React's renderer suspends on (via a custom `Suspense` fallback that reads `arguments[0]`). Assert pointer equality across re-renders for each resource's loading episode.
- **Type fixture: `promise` field is required, not optional.** `tsd` assertion: `Extract<ResourceState<unknown>, { state: 'loading' }>['promise']` resolves to `Promise<unknown>`, not `Promise<unknown> | undefined`. A PR that changes the `loading` arm to `promise?: Promise<unknown>` fails `tsc --noEmit`.
- **Type fixture: `promise` is reachable from the union without narrowing through `unknown`.** `tsd` assertion that `Extract<ResourceState<T>, { state: 'loading' }>` is a fully-typed structural shape, and that the `promise` field's type is `Promise<unknown>` (not `Promise<T>` — Theorem 3 deliberately quantifies the promise as `unknown` because the loaded value is not yet available at the loading arm).
- **Counterexample: WeakMap-cached Promise falsifies the test.** A fixture implementation that stores the Promise in a `WeakMap<Node, Promise<unknown>>` outside the union arm and reads from the WeakMap on each observation produces a different Promise after a WeakMap eviction; the property test on this implementation fails within 50 trials with a deterministic seed.

#### 5 concerns

1. **Pointer equality across reads in the same episode.** The bi-implication is `(s1.state === 'loading' ∧ s2.state === 'loading' ∧ s1.origin === s2.origin) ⇒ s1.promise === s2.promise`. The test uses `Object.is`, not `==` or `===` on a `.then`-wrapped value — pointer identity is the contract.
2. **Episode boundary — re-fetch from Loading produces a new Promise; the test asserts pointer-inequality.** This is the dual of concern 1 and the test asserts both directions: identity within the episode, distinctness across the boundary. A draft that re-uses the previous Promise across episodes (for "efficiency") falsifies the second direction.
3. **React renderer integration — SuspenseList sees the same Promise across re-renders.** The companion test is the only one in this EPIC that mounts a renderer; it closes §9.1 row 17. The test uses `@testing-library/react`'s `render` plus a custom `Suspense` fallback that captures the suspended Promise via the React internals' `__SECRET_INTERNALS_DO_NOT_USE` channel — pinned to the React major version we ship against.
4. **Type fixture — the `promise` field is required, not optional. Eliding it is a `tsc` error.** The fixture is the type-system gate; the runtime property test is the safety net for shapes that pass type-check.
5. **No race condition — controlled microtask ordering via `Promise.resolve()` chains.** Every microtask boundary in the test is owned by the test body; no `setTimeout`, no `queueMicrotask` outside an `await Promise.resolve()` flush. Brendan's reading: every microtask boundary is a chance for a theorem to fail, and the test author owns every boundary.

### TASK 10.4 — Theorem 4 property test

> **Divergence (as of v0.9.0):** The Theorem 4 witness ultimately shipped as TWO files, after a #575 reclassification: `theorem-4-behavior-domain.test.ts` (the actual behavior-domain witness, added in PR #575/#991) and `theorem-4-graphtime-monotonicity.test.ts` (kept as a supporting-lemma regression-witness — it was originally mis-named as the Theorem 4 file pre-#575). The fast-check property side of TASK 10.4 ships under EPIC-9 as `packages/sync/test/properties/origin-bound-resolution.property.test.ts` and other property files in `/properties/`. The `back-dated-registration.integration.spec.ts` integration test cited below did NOT land — the back-dated-registration scenario remains hypothetical without TASK 10.5's Rust rule.

**Files:** `packages/sync/test/theorems/behavior-domain.spec.ts` (new).

Per §3.1 Theorem 4. The test generates a graph with K ∈ [1, 16] resources registered at randomised GraphTimes `[t_1, ..., t_K]`, drives an arbitrary commit sequence of length L ∈ [0, 64], then asserts for every resource `i` and every GraphTime `t < t_i` in the retention window that `graph.readAt(node_i, t)` returns the `evicted` arm with `oldestRetainedTime === t_i`. The complementary direction — for every `t ∈ [t_i, ∞) ∩ retentionWindow`, the read returns the `retained` arm — is also asserted, so the disjunction is checked end-to-end.

#### TDD test suite

- **Property test: domain bound — every read at `t < registrationTime` is `evicted`.** 1000 trials of K resources at randomised registration times; for each trial, sample 8 random GraphTimes `t < t_i` per resource and assert `graph.readAt(node_i, t).arm === 'evicted'`.
- **Property test: recovery breadcrumb — `oldestRetainedTime === registrationTime`.** For each `evicted` result above, assert `oldestRetainedTime === t_i`. The breadcrumb is the contract; a draft that returns `oldestRetainedTime: 0` (the t₀ marker) on an `evicted` result fails this assertion.
- **Property test: complementary direction — every read at `t >= registrationTime` is `retained`.** For each resource `i` and each GraphTime `t ∈ [t_i, ∞) ∩ retentionWindow`, assert `graph.readAt(node_i, t).arm === 'retained'` and the carried `value` is a valid `ResourceState<T>`.
- **Counterexample: back-dated registration falsifies both Theorem 2 and Theorem 4.** A fixture that registers a resource via `(graph as any)._entries.set(...)` (the side-channel mutation pattern) at GraphTime `0` is wired into the integration test; the test asserts (a) the `Schema 3` IR pass refuses the export (TASK 10.5's Rust rule, once it lands), and (b) the Theorem 4 property test fails on the first trial that reads the back-dated node at `t = 0` — the read returns `retained` when the predicate demands `evicted` for `t < t_i = 0`. The Theorem 2 falsification (the side-channel mutation) is asserted by TASK 10.2's runtime counter; the Theorem 4 falsification (the domain claim now lies) is asserted here.
- **Edge: `t = registrationTime - 1` is `evicted` even when the engine clock once equalled that value.** The test specifically constructs a scenario where the engine clock advanced past `t_i` before the resource was registered, then reads `node_i` at `t_i - 1`. The result is `evicted`, not `retained` carrying some prior value — the node did not exist as an entity at that moment, and the predicate is total over the GraphTime line.

#### 5 concerns

1. **Floor: 1000 trials.** Registered via `propertyTrials.register('§3.1.T4', { numRuns: 1000 })`; same floor enforcement as Theorems 1-3.
2. **Domain bound — every read at `t < registrationTime` is `evicted`.** The property test asserts the bi-implication directly; no third arm of `RetentionResult<T>`, no fallback to `idle`.
3. **Recovery breadcrumb — `oldestRetainedTime` matches `registrationTime`.** The breadcrumb is the contract that lets a host UI rewind; a draft that returns `oldestRetainedTime: 0` defeats the recovery path silently. The property test asserts equality, not just presence.
4. **Counterexample fragment — the back-dated registration test asserts the Schema pass + the property both fail (wired via integration test).** The integration test is `packages/sync/test/theorems/back-dated-registration.integration.spec.ts` (also new under this task), which composes the property test and the Rust rule's expected refusal into a single end-to-end assertion. Until TASK 10.5 lands, the integration test runs only the property side and `it.todo`s the Rust side.
5. **No race condition — deterministic.** No loaders, no microtasks; the test exercises `graph.readAt` directly against a deterministically-constructed commit log. The trial is replayable from its seed alone.
