## 4. Honest verdict — this is the designed-for result, NOT a STOP

V2.3 is honest measurement. It documents that rust-ssot is **~85×
slower on the median commit** (the #1133/#1479 axis) — that is
**known, expected, and explicitly NOT a halt condition**
(V2-DESIGN §0 / §4.2 / §7 kill-criterion row). **No STOP-VERDICT is
filed for V2.3's perf numbers.**

- **#1133 STANDS.** The Rust-engine-in-WASM per-commit execution
  cost (~17 μs) is ~85× the TS engine (~0.2 μs), a property of
  *today's* WASM runtime, not the bridge architecture; #1493's
  batching provably cannot amortise the engine-exec gap (it
  amortises only the crossing tax — C.6 confirms 1/N).
- **The value prop is large-tree GC-survival (#1525), not median
  latency.** The real serde-wasm engine survives 50k nodes (3.5 s,
  1 natural major-GC) where TS-SSOT GC-destabilises (32.5 s, 43
  natural major-GC) — the 43→1 collapse, with two honest
  narrowings vs the #1518 synthetic (raw heap-slope is non-zero
  but **transient not retained**; p99.9 flattening is ~16.6× not
  ~437×). These narrowings do not weaken the #1133 / V2-DESIGN §0
  framing one bit.
- **T3 is the only true STOP** and it is green in shadow here; the
  load-bearing proof under *promotion* is the V2.4 GO/NO-GO.

V2.3 is a non-destructive measurement ticket. It changes no
production code, no SPEC text, and no default behaviour
(`engine: 'js-ssot'` remains byte-identical to dev `8405b783`).
